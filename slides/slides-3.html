Mapping CQL to Java Data Types

Mapping Data Types

Cassandra Datatypes and Java

  • Cassandra is Java based
  • Drivers map data types between Cassandra and Java

You may naturally think that since Cassandra is Java based, that Cassandra data types would simply the same types found in Java. However, Cassandra mostly abstracts away its Javaness and instead contains data types similar to their counterparts in the relational database world.

For example, Cassandra has TEXT whereas Java as String. Although these two types are fully convertible to each other, they have different names. Also, remember that you can access Cassandra via several languages, so Cassandra strives to be more generic than just using Java data types as is.

Simple Statements

Executing CQL

  • You can execute raw CQL
session.execute("SELECT * FROM t1");

At the most basic level, you can execute() CQL statements the same as you do using cqlsh or in DevCenter. execute() takes the CQL command text and creates a SimpleStatement under the hood.

Although you can execute() raw CQL as a string, we recommend only doing so in rare circumstances (for example, basic testing code just trying to retrieve a quick result set). execute()`ing a `SimpleStatement prevents you from setting all these options if necessary. Plus raw CQL strings are subject to CQL injection attacks.

Statements

Or use Statements

  • There are two main types of statements:

    • SimpleStatement
    • BoundStatement

  • Statements have options like

    • .setConsistencyLevel()
    • .enableTracing()
    • .setFetchSize()

SimpleStatement, BoundStatement, BatchStatement etc. are all subclasses of Statement. Statement tracks settings like consistency level so you can tune the consistency per statement. You can enableTracing(), setFetchSize(), etc.

For all Statement options, see: http://docs.datastax.com/en/drivers/java/3.0/com/datastax/driver/core/Statement.html

Result Sets

  • Executing a statement/query returns a ResultSet
  • Iterable collection of Row objects
  • Extract column values from a row
  • Automatically pages through the data
ResultSet rs = session.execute("SELECT * FROM t1");
Row firstRow = rs.one();
int col1 = firstRow.getInt("c1");
String col2 = firstRow.getString("c2");

execute() returns a ResultSet which is a container of Row objects, one per returned Cassandra row.

one() returns a single row, the next row, so you can call one() multiple times until it returns null.

The get() functions take a column index, but for maintainability use the overload that takes an explicit column name.

Since partitions can be large, ResultSet automatically pages the data through memory as you iterate trough it. Thus not all the data is retrieved at once from Cassandra. You can configure how aggressive paging is as well.

For further detail as to what you can do with a ResultSet, see: http://docs.datastax.com/en/drivers/java/3.0/com/datastax/driver/core/ResultSet.html

Iterating Result Sets

ResultSet rs = session.execute("SELECT * FROM names");
for(Row row : rs) {
        String fname = row.getString("fname");
        String lname = row.getString("lname");
        System.out.println("Name: " + fname + " " + lname);
}

For convenience, you can iterate through a ResultSet as well.

SimpleStatement

  • Simple statements are interpreted at runtime and may include parameter placeholders
  • You can set options on a SimpleStatement instance
Statement statement =
  new SimpleStatement("select * from t1 where c1 = 5");
Statement statement =
  new SimpleStatement("select * from t1 where c1 = ?", 5);
session.execute(statement);

In cases where you need more configuration than what execute() gives you by default, you can explicitly create your own SimpleStatement object and set your configuration operations. Then you pass your SimpleStatement instance to execute().

You have options when creating a SimpleStatement. You can use raw CQL as we did earlier with execute(), or you can add parameters and pass in their values. Notice the question mark indicates a parameter. You can have as many question marks as you like as long as you pass in their respective argument values in the same order the question marks appear in the query.

Lightweight Transactions

  • Return a single row ResultSet with an [applied] column
  • Helper method on the ResultSet - wasApplied()
  • Or you can read the [applied] column directly
INSERT INTO user(email, uname, password) VALUES
('steve@foo.com', 'steve', 0xCAFEFACE) IF NOT EXISTS;

Result:

[applied] | email          | uname  | password |
----------+----------------+--------+----------
   False  | steve@foo.com  | steve  | 0xCAFECAFE

Prepared Statements

Using Prepared Statements

  • Prepared statements are efficient
  • Compiled once on each node automatically as needed
  • Prepare each statement only once per application
  • Use one of the many bind variations to create a BoundStatement
PreparedStatement ps = session.prepare("SELECT * from t1 where c1 = ?");
BoundStatement bound = ps.bind(5);

bind() takes n number of arguments.

PreparedStatements are the next level of hammer you could use in your application. Generally you should probably prefer PreparedStatements over SimpleStatements. PreparedStatement does not inherit from Statement as you cannot execute() a PreparedStatement directly. You must first bind() the PreparedStatement to get a BoundStatement, which is execute()able.

A gain with PreparedStatements that that they are compiled once per node, unlike a SimpleStatement that is compiled once per call. So if you have a query that you run frequently, you will see performance gains using a PreparedStatement. PreparedStatements hold placeholders for your parameter values which you can change on every execution, but Cassandra will not have to re-parse and re-compile your PreparedStatement query.

If you have a 1,000 node cluster with a replication factor of 1, and you query for 1,000 individual partitions separately, and each partition resides on one node, then every node will have to compile your query. However, if you have a replication factor of three, then only 333 nodes would have to compile your query because you know all of your data is stored on 1/3 of your nodes. However, those 333 nodes would compile once and execute three times, which is a performance gain. This is if the driver is using token aware policy.

So SimpleStatements are good for occasional one-offs (convenience), and PreparedStatements are good for queries that execute frequently.

Named Parameters

No need to worry about the order of ?

PreparedStatement ps = session.prepare("INSERT INTO t1 (c1,c2,c3) VALUES(:c1, :c2, :c3)");
Statement bound = ps.bind()
                    .setInt("c1", 5)
                    .setString("c2","foo")
                    .setDate("c3", myDateVariable);

Using the question mark notation quickly becomes confusing and unmaintainable as the number of parameters grows. We recommend using explicitly named parameters and binding their argument values.

Query Builder

Query Builder API

Alternative to building string queries manually

  • Contains methods to build SELECT, UPDATE, INSERT and DELETE statements
  • Generates a Statement as per the earlier techniques

Method

Return Type

QueryBuilder.select()

Selection

QueryBuilder.insertInto()

Insert

QueryBuilder.update()

Update

QueryBuilder.delete()

Selection

If you’re not familiar with CQL, the query builder API can help ensure your query is correct. You simply build up your query by calling successive functions on the returned type. The returned type (Selection, Insert, etc.) contains only methods applicable to that type of query.

delete() returns a Selection because a DELETE needs a FROM and a WHERE, which a Selection object has. DELETE is like a SELECT except it removes the data instead of returning it.

  • http://docs.datastax.com/en/developer/java-driver/3.0/java-driver/reference/queryBuilderOverview.html
  • http://docs.datastax.com/en/drivers/java/3.0/com/datastax/driver/core/querybuilder/QueryBuilder.html

Example

Most basic use specifies the keyspace and table

Select select = QueryBuilder.select()
     .from(<keyspace_name>, <table_name>);
  • Returns all data for a given table
  • Returns a Selection object which inherits Statement
  • Thus all Statement options are available:
Statement statment = ...
statement.enableTracing();
Boolean idempotent = statement.isIdempotent();

Additional Options

Build up your query using multiple options

Select select = QueryBuilder.select()
     .all()
     .distinct()
     .from(<keyspace>, <table>)
     .where(eq(<field1>, <value1>))
     .and(eq(<field2>, <value2>))
     .and(gte(<field3>, <value3>));
  • Additional options are available
  • i.e. limit(), orderBy() and more
  • See the API doc for Select, Where and Statement

Insert Example

// Insert a user
UUID userID = UUIDs.random();
Insert insertNewUser =
    QueryBuilder
        .insertInto("killrvideo", "user")
        .value("email", "moo@you.com")
        .value("fname", "Mooer")
        .value("lname", "Youer")
        .value("password", "chicken")
        .value("user_id", userID);

System.out.println(insertNewUser.getQueryString());
// Ouput:
// INSERT INTO killrvideo.user (email,fname,lname,password,user_id) VALUES (?,?,?,?,?)

session.execute(insertNewUser);

Here we craft an INSERT of a new user into our killrvideo.user table. You can see the resulting query string from getQueryString() output.

Inserting Two Videos

// Insert two videos for the user
UUID video1Id = UUIDs.timeBased();

Insert insertVideo1 =
    QueryBuilder
        .insertInto("killrvideo", "videos_by_tag")
        .value("tag", "adventure")

        .value("description", "Awesome pirates!")

UUID video2Id = UUIDs.timeBased();

Insert insertVideo2 =
        QueryBuilder

                .value("tag", "adventure")
                .value("video_id", video2Id)
                .value("description", "More awesome pirates!")
                .value("title", "Dead Man's Chest");

SELECT Example

// Retrieve the oldest adventure video
Select selectOldestVideo =
        QueryBuilder
                .select()
                .from("killrvideo", "videos_by_tag")
                .where(eq("tag", "adventure"))
                .orderBy(asc("video_id"))
                .limit(1);

System.out.println(selectOldestVideo.getQueryString());
// Output:
// SELECT * FROM killrvideo.videos_by_tag WHERE tag=? ORDER BY video_id ASC LIMIT 1;

ResultSet oldestVideo = session.execute(selectOldestVideo);
System.out.println(oldestVideo.one().getString("title"));
// Output:
// Curse of the Black Pearl

DELETE Example

// Delete everything we just created
Delete.Where deleteUser = QueryBuilder.delete()
        .from("killrvideo", "user")
        .where(eq("email", "moo@you.com"));
Delete.Where deleteVideo1 = QueryBuilder.delete()
        .from("killrvideo", "videos_by_tag")
        .where(eq("video_id", video1Id))
        .and(eq("tag", "adventure"));
Delete.Where deleteVideo2 = QueryBuilder.delete()
        .from("killrvideo", "videos_by_tag")
        .where(eq("video_id", video2Id))
        .and(eq("tag", "adventure"));

System.out.println(deleteUser.getQueryString());
// Output:
// DELETE FROM killrvideo.user WHERE email=?;

session.execute(deleteUser);
session.execute(deleteVideo1);
session.execute(deleteVideo2);

Batches

Executing Batches

Two methods for executing a batch

  • Using the CQL batch syntax in the actual code
  • Creating a BatchStatement which contains substatements

Batch with Raw CQL

...
String batch = "BEGIN BATCH " +
               "INSERT INTO ... " +
               "UPDATE ... " +
               "DELETE ..." +
               "APPLY BATCH";
session.execute(batch);

You can simply encode your regular CQL batch statements into a string. Don’t.

Using a BatchStatement

  • You can use QueryBuilder to create Statements
  • You can also bind your PreparedStatement when adding to the BatchStatement
...
BatchStatement batch = new BatchStatement();
batch.add(otherStatement);
batch.add(anotherStatement);
...
session.execute(batch)
